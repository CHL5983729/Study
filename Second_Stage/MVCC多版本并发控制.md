# MVCC多版本并发控制

## 1.什么是MVCC

MVCC全称：“多版本并发控制器”，通过为每个数据行维护多个历史版本，让读操作和写操作不互相阻塞，从而大幅度提升数据库的并发性能

## 2.为什么需要MVCC

在传统数据库中，并发控制主要依靠锁：

                读锁（共享锁）：多个事物可以同时读，但是写必须等待

                写锁（排他锁）：写的时候，其他事物既不能读也不能写

这样虽然能保证数据一致性，但读写严重冲突，并发性能很差。

MVCC的解决思路是：每个写操作生成一个新版本，读操作可以读取一个旧版本，从而避免冲突

## 3.核心概念

#### 3.1 事物ID

每个事物启动时，数据库会分配一个全局唯一递增的ID，ID越大事物越新

#### 3.2 行版本与隐藏字段

在InnoDB中，每一行数据实际上包含两个隐藏的字段：

                DB_TRX_ID：最近修改该行的事物ID

                DB_ROLL-PTR：指向该行上一个版本的回滚指针，通过undo log链可以找到历史版本

#### 3.3 undo log （回滚日志）

每次修改数据，InnoDB都会记录一条undo log，记录了修改前的数据内容，用于回滚或构造历史版本

#### 3.4 ReadView （读视图）

ReadView是事物进行快照读时生成的“快照”，它决定了该事物能“看到”哪个版本的数据

ReadView 包含以下信息：

                m_ids：当前系统中所有活跃（尚未提交）的事务 ID 列表

                min_trx_id：活跃事务中的最小 ID

                max_trx_id：下一个将要分配的事务 ID（可理解为一个上限）

                creator_trx_id ：创建该 ReadView 的事务的 ID

## 4.MVCC解决的问题与存在的问题

#### 4.1 解决的问题

读写不阻塞：读操作无需等待写锁，直接读取旧版本；写操作无需等待读锁，直接生成新版本

提高并发：尤其适合读多写少的场景

实现事物的隔离级别：读已提交，可重复度

#### 4.2 存在的问题

储存开销：每个修改都要保留旧版本数据，占用undo log空间

清理：数据库后台需要定期清理不在被任何事物需要的旧版本，否则undo表空间会暴涨

当前读仍然需要锁：MVCC只对快照读有效，当前读依然会锁行，可能引起死锁

写写冲突仍需串行：两个事物同时修改一行数据，后发生的写事物会阻塞或者回滚
