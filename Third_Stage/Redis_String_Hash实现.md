# Redis 底层String Hash 实现原理

## 1.Redis最核心的设计思想

Redis不使用一种固定的方式存储数据，而是会根据数据大小和类型自动选择存储方式。

数据小的时候：怎么省内存怎么来；数据大的时候：怎么快怎么来

而且这个转换是自动，单向，不可逆的

## 2.String类型

String是Redis最简单的类型，一个Key对应一个Value，但其背后有3种完全不同的存储方式

Redis中存储的每一个数据，不管什么类型，都有一个16字节的RedisObject，记录了这个数据的基本信息：

#### 2.2.Redis的字符串：

Redis没有使用普通的字符串，而是自己发明了一种 ”智能字符串“ ：SDS

优点：自动获取长度，自动加大小，可以存任何东西

#### 2.3.String的存储三种方式

Redis会根据你存储的内容，自动选择下面三种方式中的一种：

1.int编码

适用情况：存的是一个整数

特点：最省内存，最快，支持原子操作

2.embstr编码

适用情况：存的是字符串，且长度<= 44字节

特点：只需要一次申请内存，一次释放内存，读取速度快（在内存中是连续的）

为什么是44字节：

- RedisObject占 16 字节
- 智能字符串头占 3 字节
- 末尾空白页占 1 字节
- 总共 20 字节的固定开销
- Redis 每次申请内存最小是 64 字节
- 64-20=44 字节，正好是能存的最大内容

3.raw编码

适用情况：字符串长度>44字节

特点：需要两次申请，释放内存，读起来稍慢（在内存中是分开的），适合长字符串

#### 2.4.编码转换规则

String的编码只能升级，不能降级

int编码（当把数字改成字符串）---> embstr编码（当字符串长度超过44字节）---> raw编码

即使后来把长字符串改回短字符串，编码也不会变回去

## 3.Hash类型

Hash类型就像是一个小字典，里面可以存多个键值对，特别适合存储用户信息，商品属性这类对象

#### 3.1.Hash的两种存储方式

Hash有两种存储方式，Redis会自动选择：

1.紧凑型（listpack）：小数据量时使用，极省内存

2.带索引的大书架（hashtable）：大数据量时使用，极快

方式一（紧凑型）：

适用情况：字典中键值对数量 <= 512个，所有键和值的长度都 <= 64字节

特点：超级省内存，查找速度快（因为只有512个键值对）

方式二 （带索引的大书架）：

适用情况：只要不满足方式一的任意一个条件，就回自动转换

特点：查找速度超级快，比较费内存

#### 3.2.编码转换规则

和String一样，Hash的编码转换也是不可逆的

紧凑型（listpack）（字段数>512 或 任意值>64字节）---> 带索引的大书架（hashtable）

即使后来把大字段删了，或者把字段数减少到 512 以下，编码也不会变回去
