# Redis RDB与AOF持久化

Redis是内存数据，所有数据存储在内存中，一旦退出或者断电，数据就会消失，持久化就是将数据异步保存到磁盘中，重启时在从磁盘中恢复数据的机制

## 1.RDB持久化

#### 1.1.什么是RDB

RDB是Redis默认开启的持久化方式，它会在指定时间间隔，将内存中所有数据生成一份二级制格式的全量快照，并且保存到磁盘上

核心特点：记录某个时间点的完整数据状态，类似与MySQL的全量备份

#### 1.2.RDB工作原理

RDB的高性能得益于操作系统的写时复刻技术，整个过程几乎不受阻塞线程

- **触发快照**：满足触发条件时，Redis 主进程调用`fork()`系统调用创建子进程
- **共享内存**：父子进程**共享同一块物理内存**，子进程看到的是 fork 瞬间的内存状态
- **写时复制**：如果父进程修改数据，操作系统会复制对应的**4KB 内存页**，父进程修改副本，子进程继续读取原数据
- **生成快照**：子进程遍历共享内存，将数据写入临时 RDB 文件
- **原子替换**：子进程完成后，用临时文件**原子替换**旧的 RDB 文件，然后退出

#### 1.3.RDB触发方式

手动触发：

                SAVE：阻塞主进程，直到快照完成（生产环境不能使用）

                BGSAVE：后台异步执行，fork子进程完成快照

自动触发：

                在redis.conf中配置save参数，格式为save<秒数> <修改次数>，满足任意一个条件就会自动触发BGSAVE

```conf
                # 默认配置（满足任意一个就触发）
save 900 1      # 15分钟内至少1次写操作
save 300 10     # 5分钟内至少10次写操作
save 60 10000   # 1分钟内至少10000次写操作

# 关闭自动RDB（纯缓存场景可考虑）
# save ""
```

#### 1.4.RDB优缺点

优点：

恢复速度快：二进制格式，直接加载到内存，比AOF快10倍以上

文件体积小：压缩后的二级制文件，通常只有AOF的十分之一大小

对性能影响小：仅在快照时fork子进程，日常运行无额外开销

适合备份：可定期复制RDB文件到远程存储，用于备份

缺点：

数据丢失风险大：两次快照之间的数据会丢失

fork阻塞问题：大数据集时，fork操作可能阻塞主线程几十到几百ms

不适合实时持久化：频繁快照会严重影响性能

## 2.AOF持久化

#### 2.1.什么是AOF

AOF是另一种持久化方式，它会记录Redis执行的所有写命令，以文本格式追加到文件末尾，重启时，通过执行AOF文件中的所有命令来恢复数据

核心特点：记录所有修改数据的操作，类似于MySQL的binlog

#### 2.2.AOF工作原理

- **命令执行**：Redis 执行写命令后，将命令追加到**AOF 缓冲区**
- **刷盘**：根据配置的刷盘策略，将缓冲区中的数据写入磁盘
- **重写**：当 AOF 文件过大时，自动执行重写操作，压缩文件体积

#### 2.3.AOF刷盘策略

appendfsync参数直接决定了数据安全性和性能的平衡，有三个选项：

```conf
# 每次写命令都执行fsync（最慢，最安全）
# 数据零丢失，但性能极差，只适合金融核心场景
appendfsync always

# 每秒执行一次fsync（默认，强烈推荐）
# 性能接近Redis原生速度，最多丢失1秒数据
appendfsync everysec

# 由操作系统决定何时刷盘（最快，最不安全）
# 可能丢失几秒到几十秒的数据，绝对不推荐生产使用
appendfsync no
```

#### 2.4.AOF重写

为什么要重写：AOF文件会不断变大，比如对同一个Key执行100次操作，AOF会记录100条命令，但最终结果只需要最后一条，重写可以将多个命令合并成一个，大幅减小文件体积

| 文件类型        | 数量     | 内容                        |
| ----------- | ------ | ------------------------- |
| BASE 文件     | 最多 1 个 | 重写时生成的**全量快照**（RDB 格式）    |
| INCR 文件     | 多个     | 记录重写开始后的**增量写命令**（AOF 格式） |
| MANIFEST 清单 | 1 个    | 记录当前 AOF 由哪些文件组成          |

重写流程：

1.fork子进程，子进程生成新的BASE文件

2.主进程打开一个新的INCR文件，新命令直接写入这个文件

3.子进程完成BASE文件后，主进程更新MANIFEST清单

4.原子切换到新的文件集合，清理旧文件

#### 2.5.AOF的优缺点

优点：

数据安全性高：everysec策略最多丢失1秒数据，always策略零丢失

实时性好：可以做到秒极持久化

文件可读性高：文本格式，可以直接查看和修改

缺点：

恢复速度慢：需要执行命令，大数据集可能需要几分钟

文件体积大：记录所有命令，比RDB大很多

性能影响比RDB大：每秒都要执行fsync，会有一定的性能开销

## 3.混合持久化

#### 3.1.什么是混合持久化

Redis中混合持久化结合了RDB和AOF的所有优点

原理：AOF的文件前半部分是RDB格式的全量快照（占99%数据），后半部分是AOF格式的增量命令

恢复流程：先快速加载RDB部分 -----> 在执行少量增量命令

优势：恢复速度接近RDB，数据安全性接近AOF

```conf
# 开启AOF
appendonly yes
# 开启混合持久化（默认yes，无需额外配置）
aof-use-rdb-preamble yes
# Redis 7.0+自动启用MP-AOF
```

| 对比维度   | RDB         | AOF（everysec） | 混合持久化（推荐）      |
| ------ | ----------- | ------------- | -------------- |
| 数据丢失窗口 | 几分钟         | 1 秒           | 1 秒            |
| 恢复速度   | 极快          | 慢             | 接近 RDB（快 10 倍） |
| 文件体积   | 小           | 大             | 中等             |
| 性能影响   | 小           | 中等            | 中等             |
| 内存开销   | 小（fork 时）   | 大（旧版重写时）      | 小（MP-AOF）      |
| 运维复杂度  | 低           | 中等            | 低              |
| 适用场景   | 纯缓存，可容忍数据丢失 | 数据安全要求高       | 所有企业级业务        |
